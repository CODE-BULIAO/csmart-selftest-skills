# 巡火测试通用原则

## 适用范围

本文件定义适用于所有巡火项目的通用测试原则。具体的点位、触发词、调度时间、群 ID 等项目数据从 `project-profile.json` 获取，不在此文件中固定。

执行测试前必须读取最新需求文档，本文件保存稳定回归规则，不代替最新需求。

## 通用测试类别

| 类别 | 测试目的 | 设计原因 |
|---|---|---|
| 测试群隔离 | 验证只向测试群发送，拒绝生产群和其他业务群 | 错发生产群的影响高于普通功能缺陷，必须最先阻断 |
| 点位字典完整性 | 验证运行时点位配置覆盖所有区域和点位 | 点位缺失或错序会使全部统计结果失真 |
| 触发词识别 | 验证所有配置的触发词均能启动解析流程 | 触发词是消息进入解析流程的入口 |
| 单点位上报 | 验证单个点位可被正确标记和归属 | 单点位是区间展开的基础单元 |
| 正向区间展开 | 验证从起点到终点的连续区间展开 | 标准连续区间是核心业务逻辑 |
| 反向区间展开 | 验证反向区间与正向区间结果一致 | 验证 min/max 索引算法的正确性 |
| 连接符归一化 | 验证多种连接符（`~`、`～`、`—`、`——`）归一化为 `-` | 现场输入不统一，归一化是最常见的解析失败来源 |
| 空格和逗号处理 | 验证空格不影响区间，中英文逗号分隔的点位分别处理 | 覆盖常见输入格式差异 |
| 特殊点位 | 验证非纯数字点位（如 `AM`、`CUR`、`BM1`）可独立标记且归属正确 | 特殊点位不能依赖纯数字楼层解析算法 |
| 非法输入拒绝 | 验证不存在的点位、空输入、格式错误不污染状态 | 避免错误输入产生虚假已巡记录 |
| 自然语言引导 | 验证自然语言输入不误识别，并给出格式引导 | 覆盖设计文档通常列出的输入风险 |
| 总结正确性 | 验证总结消息正确显示已巡（✅）和缺失（❌）点 | 总结是用户最终使用的核心输出 |
| 自动总结调度 | 验证配置的 cron 时间均有实际执行证据 | 配置存在不能证明消息成功推送 |
| 手动总结兜底 | 验证手动触发词立即生成总结 | 作为 WhatsApp 消息漏抓或调度失败时的人工兜底 |
| 跨业务日重置 | 验证零点后状态重置或切换，不混入前一天记录 | 防止跨日数据污染 |
| 重复上报幂等 | 验证同一点位重复上报不导致统计虚高 | 原始需求通常规定每日一次，避免重复照片造成统计错误 |

## 动态数据获取

以下数据从 `project-profile.json` 获取：

- **点位字典**：`requirements.zones` 定义所有区域和点位
- **触发词**：`requirements.trigger_words` 列出所有触发词
- **连接符**：`requirements.separators` 列出支持的连接符
- **手动总结词**：`requirements.manual_summary_words` 列出触发词
- **调度时间**：`requirements.summary_schedule` 定义 cron 表达式
- **跨日规则**：`requirements.business_day_rule` 定义业务日起始时间

## 新需求测试规则

读取新需求后，逐条记录"旧规则、新规则、影响范围、兼容风险"。每条变化至少生成：

1. 正常用例：证明新增功能可以使用
2. 边界用例：覆盖时间、区间端点、空值或最大集合
3. 异常用例：证明无效输入不会污染状态
4. 兼容用例：当新规则可能影响旧输入时，验证原有通用契约仍通过

只有已确认上线且长期有效的规则才进入固定集；临时需求和探索性功能保留在当次 `new-cases.jsonl`。

## 证据要求

每条 live 用例至少保留两个证据层：发送或回复、业务日志、前后状态快照、定时任务历史。总结测试同时核对配置时间和实际推送记录。
